```tsx
            <Button type="submit" className="w-full" variant="outline">
              Continue with Google
            </Button>
          </form>

          <div className="relative">

            <div className="relative flex justify-center text-xs uppercase">
              <span className="bg-background px-2 text-muted-foreground">or</span>
            </div>
          </div>

          <form
            action={async (formData: FormData) => {
              "use server";
              const email = formData.get("email");
              if (typeof email !== "string") return;
              await signIn("resend", { email, redirectTo: "/dashboard" });
            }}
            className="flex flex-col gap-3"
          >
            <Label htmlFor="email">Email</Label>
            <Input id="email" name="email" type="email" required placeholder="you@example.com" />

          </form>
        </CardContent>
      </Card>
    </main>
  );
}
```
Notes:
- The `"use server"` directive INSIDE the form action makes it a Server Action — works with RSC.
- `redirectTo: "/dashboard"` lands the user there after successful auth.
- The `(auth)` route group lets us add layout decoration later without changing URLs.

4. Create `app/(app)/dashboard/page.tsx`. EXACT content:
```tsx
import { auth } from "@/lib/auth";
import { redirect } from "next/navigation";
import { Card, CardContent, CardHeader, CardTitle } from "@/components/ui/card";
import { SignOutButton } from "@/components/sign-out-button";

export default async function DashboardPage() {
  const session = await auth();
  if (!session?.user) redirect("/signin");

  return (
    <main className="mx-auto flex min-h-screen max-w-3xl flex-col gap-6 p-8">
      <header className="flex items-center justify-between">
        <h1 className="text-2xl font-semibold">Welcome back, {session.user.name ?? session.user.email}</h1>
        <SignOutButton />

        <CardHeader>
          <CardTitle>Dashboard placeholder</CardTitle>
        </CardHeader>
        <CardContent className="text-muted-foreground">
          Phase 1 is live. Phase 2 will add transactions and categorization.
        </CardContent>
      </Card>
    </main>
  );
}
```
Notes:
- Defense in depth: middleware (Plan 03) ALSO protects `/dashboard/*`, but this page re-checks `auth()` so direct navigation, dev-bypassed middleware, or future config drift can't expose data.

- `session.user.name ?? session.user.email`: Google OAuth users get `name`, Resend magic-link users may not — fall back to email.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Auth.js handler | OAuth callback + magic-link verify cross from browser to /api/auth/[...nextauth] |
| Form action -> Auth.js signIn | RSC form action transitions to server, calls signIn server-side |
| Server -> Google / Resend | OAuth code exchange and email send originate from server, not client |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04-01 | Spoofing | sign-in form CSRF | mitigate | Auth.js v5 server actions are scoped to the originating session; Auth.js applies CSRF tokens internally per its default. AUTH_TRUST_HOST=true is set for Vercel hosting. |
| T-01-04-02 | Tampering | callbackUrl manipulation | mitigate | `redirectTo: "/dashboard"` is hardcoded in the form actions (server-side), not user-controlled. Middleware-driven `callbackUrl` query param is set by middleware itself in Plan 03 from `req.nextUrl.pathname` (server-side). Auth.js v5 default rejects external redirect targets. |
| T-01-04-03 | Information Disclosure | session.user fields in JSX | mitigate | Only `name` and `email` are rendered. No PII or tokens leak to client. |
| T-01-04-04 | Elevation of Privilege | bypass middleware | mitigate | dashboard/page.tsx independently calls `auth()` and `redirect("/signin")` — defense in depth. |
| T-01-04-05 | Information Disclosure | Resend magic-link reveals email | accept | Inherent to magic-link UX. Auth.js v5 implements anti-enumeration by always responding identically whether email exists or not. |
| T-01-04-06 | Tampering | sign-out request | mitigate | next-auth/react signOut performs CSRF-protected POST to /api/auth/signout. Standard Auth.js behavior. |
| T-01-04-07 | Denial of Service | unbounded magic-link requests | accept | Resend free tier rate-limits at provider side. Auth.js verificationToken table bounded by row inserts (cleaned up periodically). For Phase 1, accept the risk. |
| T-01-04-08 | Information Disclosure | error messages on signIn failure | accept | Auth.js default error pages don't leak internal details. Custom error UI is out of scope for Phase 1. |
</threat_model>
